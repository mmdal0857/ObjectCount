# 다중 핸드폰 카메라 기반 컨베이어 오브젝트 카운팅 시스템 — 설계

- 날짜: 2026-07-12
- 상태: 승인됨 (브레인스토밍 세션에서 섹션별 승인 완료)

## 1. 개요와 사업 배경

고가의 산업용 비전 카메라 시스템 대신 **다수의 중고 안드로이드 핸드폰**을 카메라로 활용하여,
컨베이어 레일 위를 지나가는 사전 학습된 오브젝트를 카운팅하는 시스템.
저비용 하드웨어가 사업의 핵심 경쟁력이며, 소프트웨어가 하드웨어의 한계를 보완한다.

### 확정된 요구사항

| 항목 | 결정 |
|---|---|
| 촬영 | 고정 위치, 줌/포커스 제어 가능, 1초당 5프레임 |
| 카메라 배치 | 라인당 1대 기본 + 한 라인 다중 폰도 지원 (혼합) |
| 오브젝트 등록 | 본사/관리자 사전 등록 (당일 리드타임 허용, 수시 변경) |
| 인프라 | 현장 PC/미니서버(추론·집계) + 폰(캡처 전용) |
| 폰 기기 | 중고 안드로이드 통일 |
| 물체 흐름 | 단일 품종 · 간격 통과 기본, 붙음/겹침 발생 가능 |
| 결과 소비 | 현장 실시간 대시보드 우선, ERP/MES 연동은 후순위 |
| 정확도 | 참고용(±2~3%)으로 시작 → 정산용(무오차 지향)으로 단계 개선 |

### 검토한 접근법과 선택

- **A. 기성 IP카메라 앱 활용** — 개발 최소·PoC 최속이나 줌/포커스 잠금·원격관리·자동복구 불가. 기각.
- **B. 자체 캡처 앱 + PC 추론 (선택)** — 폰은 "잘 길들여진 카메라", 지능은 전부 PC.
  오브젝트 교체 시 PC 모델 파일만 교체, 폰 앱 불변. 중고폰 성능 편차 무관.
- **C. 온디바이스 추론** — 네트워크 부하 최소이나 모델 배포 복잡·기종별 튜닝·디버깅 난도로
  "수시 오브젝트 교체" 요구와 상성 나쁨. 기각.

## 2. 시스템 아키텍처

```
[현장]                                          [본사]
 폰1 캡처앱 ─┐                                   관리 콘솔(등록 파이프라인)
 폰2 캡처앱 ─┼─ Wi-Fi(JPEG 5fps+메타) ─▶ 엣지 서버   │
 폰N 캡처앱 ─┘                          (현장 PC)  │ 모델 패키지 배포
      ▲                                  │  ▲______│   (인터넷 pull 또는 USB)
      └── 원격 설정(줌/포커스/ROI) ────────┘
                                         ▼
                                   대시보드(웹) + 카운트 DB(SQLite)
```

4개의 독립 유닛. 각 유닛은 한 가지 책임만 갖고 명확한 인터페이스로만 통신한다.

### ① 캡처 앱 (Android / Kotlin / CameraX)

- 촬영 파라미터 고정: 줌 배율, 수동 포커스 거리, 노출·화이트밸런스 잠금
  (프레임 간 화질 일관성 = 감지 정확도의 토대)
- 5fps JPEG 캡처 → 엣지 서버로 WebSocket 전송 (타임스탬프·기기ID·시퀀스 번호 메타 포함)
- 하트비트(배터리·온도·실측 fps) 송신, 서버로부터 원격 설정 수신
- 부팅 시 자동 실행(키오스크), 화면 꺼짐 방지, 상시 전원 전제
- 수집 모드: 카운팅 전송 대신 로컬 녹화 (학습 데이터 수집용, §4-①)
- **하지 않는 것: 추론, 카운팅** — 폰 교체·추가는 앱 설치 + 서버 주소 입력이 전부

### ② 엣지 서버 (현장 PC / Python)

- 수신 게이트웨이: 카메라 N대의 프레임 스트림 수신·버퍼링
- 카메라별 추론 워커: 감지 → 추적 → 가상선 카운팅 (§3)
- 카운트 이벤트 로컬 DB 기록, 대시보드 서빙 — **인터넷 없이 현장 단독 완결**

### ③ 등록 파이프라인 (본사)

- 샘플 수집 → 반자동 라벨링 → 파인튜닝 → 패키징 → 배포 (§4)
- MVP는 스크립트+체크리스트 수준에서 시작, 점진 자동화

### ④ 대시보드 (웹, 엣지 서버가 서빙)

- 라인별 실시간 카운트, 추이, 카메라 상태, 설정(ROI/카운트라인/품종 전환), 세션 히스토리

**핵심 원칙**: 오브젝트가 바뀌면 ②의 모델 파일만 교체된다. ①·④는 불변.

## 3. 카운팅 파이프라인

```
프레임 수신 → ROI 크롭 → 감지(YOLO) → 추적(ByteTrack) → 가상선 통과 판정 → 카운트 이벤트
```

1. **ROI**: 대시보드에서 카메라별 벨트 영역 폴리곤 지정. 배경 오탐 차단 + 추론 입력 축소.
2. **감지**: 파인튜닝된 YOLO nano/small. 단일 품종 전제로 활성 클래스 1개.
   엔트리급 GPU(RTX 3050급)로 카메라 8대 × 5fps = 초당 40추론 처리 가능.
3. **추적**: ByteTrack — 저신뢰 감지 박스까지 트랙 연결에 활용, 5fps의 큰 프레임 간
   이동량에 강함. 컨베이어의 일정한 이동 방향을 모션 예측 프라이어로 활용해 ID 스위칭 최소화.
4. **카운트 판정**: 벨트 진행 방향에 수직인 가상 카운트라인. 트랙 중심점이 정방향으로
   넘는 순간 +1. 트랙 ID 유지 중 재카운트 없음. 역방향 통과는 무시/차감 정책 설정 가능.
5. **붙음·겹침 대응 (단계적)**
   - 1차: 학습 데이터에 붙은 케이스 포함, 개별 인스턴스 분리 학습 (라벨링 지침, §4-②)
   - 2차: 트랙 박스 면적 > 품종 평균 면적 × 임계배수(기본 1.8)이면 **의심 이벤트**
     플래깅 + 스냅샷 저장. 참고용 단계에선 1개로 카운트하되 대시보드에 의심 건수 별도 표시.
   - 정산용 단계: 의심 이벤트만 보조 카메라 교차 검증 + 사람 재검수 UI로 확정 (Phase 3)
6. **한 라인 다중 폰 융합 (MVP 규칙)**: **주(primary) 카메라 1대가 카운트의 단일 진실.**
   보조(aux) 카메라는 의심 이벤트 시 다른 각도 스냅샷 증빙만 남긴다.
   완전한 다중 뷰 실시간 융합은 Phase 3.
7. **5fps 물리 제약 가드**: 추적 성립 조건 = 물체가 시야 통과 중 최소 3프레임 포착.
   즉 `시야 폭(m) / 벨트 속도(m/s) ≥ 0.6초`. 설치 가이드에 공식 명시
   (예: 벨트 1m/s → 시야 폭 60cm 이상을 폰 높이·줌으로 확보).
   대시보드에 "물체당 평균 포착 프레임 수" 진단 표시로 설치 불량 즉시 발견.
8. **시각 동기화**: 서버 수신 시각 단일 기준. 폰 시계 동기화 불필요.

## 4. 오브젝트 등록·모델 배포 (본사 워크플로우)

1. **샘플 수집** — 캡처 앱 수집 모드로 신제품을 실제 컨베이어(또는 유사 리그)에
   1~3분 흘리며 촬영 → 수백 프레임 확보. **현장과 같은 폰·각도·조명으로 학습 데이터를
   만들어 도메인 갭 제거.** 붙은 케이스를 의도적으로 연출해 포함.
2. **반자동 라벨링** — 기존 모델(또는 SAM 계열)이 박스 자동 제안, 사람은 수정만.
   라벨링 지침: 붙은 물체는 개별 박스, 프레임 가장자리 잘린 물체 포함.
3. **파인튜닝** — 표준화된 학습 스크립트 하나로 고정(YOLO 전이학습). 검증 셋 자동 분리.
   **품질 게이트: 검증 성능 기준(초기값 mAP50 ≥ 0.95, 운영하며 조정) 미달 시 배포 차단.**
   관리자는 ML 지식 없이 "샘플 넣고 실행" 수준을 목표.
4. **모델 패키지** — 가중치 + 메타데이터(품종 ID, 평균 물체 면적 프라이어, 학습일,
   검증 성능)를 버저닝해 패키징. 면적 프라이어는 §3-5 의심 판정에 자동 사용.
5. **배포·전환** — 엣지 서버가 패키지 수신(인터넷 pull 또는 USB 반입 — 폐쇄망 대비).
   대시보드에서 라인별 활성 품종 선택 → 무중단 모델 스왑 + 새 카운트 세션 시작.
   이전 품종 카운트는 세션 단위 아카이브.

리드타임 목표: 수집 30분 + 라벨링 1~2시간 + 학습 1시간 이내 = **당일 등록 완료**.

## 5. 데이터 모델 (엣지 로컬 SQLite)

```
lines(id, name, active_product_id, active_session_id)
cameras(id, line_id, role[primary|aux], config_json, last_heartbeat)
products(id, name, model_version, area_prior)
count_sessions(id, line_id, product_id, started_at, ended_at)
count_events(id, session_id, camera_id, ts, direction, suspect_flag, snapshot_path)
```

- 카운트는 **이벤트 단위 즉시 커밋** — 서버 재시작에도 카운트 유실 없음.
- 시간대별 집계는 이벤트 테이블 쿼리로 파생.
- ERP/MES 연동(후순위)은 이 테이블을 읽는 어댑터 추가로 대응.

## 6. 대시보드 화면

1. 라인 카드: 실시간 카운트 · 분당 처리율 · 의심 건수
2. 카메라 상태: 온라인 여부 · 실측 fps · 배터리 · 발열 · 최근 프레임 미리보기
3. 설정: 카메라 프레임 위 ROI 폴리곤/카운트라인 드로잉, 품종 전환
4. 세션 히스토리 · 의심 이벤트 스냅샷 열람

## 7. 운영·장애 처리

- 폰 하트비트 10초 단절 → 대시보드 경보 + 해당 라인 "신뢰불가" 구간 표시
- 프레임 시퀀스 번호로 유실률 측정·노출 — Wi-Fi 품질 문제를 데이터로 진단
- 폰 발열 시 앱이 fps 자동 강하 후 경고, 충전 상한 설정으로 배터리 보호
- 서버 재시작 복원력: 이벤트 즉시 커밋(§5)

## 8. 테스트 전략

- **리플레이 하네스 (핵심 자산)**: 실제 컨베이어 녹화 영상 + 사람이 센 정답 카운트 쌍을
  축적. 파이프라인 변경 시마다 전체 영상 셋 재생 → 정확도 회귀 검사.
  참고용→정산용 개선이 수치로 진행된다.
- 유닛 테스트: 카운트라인 판정, 면적 이상치 판정, 융합 규칙
- 통합 테스트: 가짜 폰 시뮬레이터(녹화 영상을 5fps WebSocket 전송)로 E2E

| 폰 앱 | Kotlin, CameraX, 포그라운드 서비스, WebSocket |

## 10. 단계별 로드맵

- **Phase 1 — 참고용 MVP**: 단일 라인, 주 카메라 카운팅, 등록 파이프라인(수동 스크립트),
  대시보드, 리플레이 하네스
- **Phase 2 — 운영 강화**: 다중 라인, 보조 카메라 증빙 스냅샷, 의심 이벤트 재검수 UI,
  클라우드 집계
- **Phase 3 — 정산용**: 다각도 교차 검증, 카운트별 증빙 클립, 감사 로그, ERP/MES 어댑터
